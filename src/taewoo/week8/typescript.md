# TypeScript 정리

## 1. TypeScript의 설명
TypeScript(이하 TS): 마이크로소프트에서 출시한 프로그래밍 언어로 JavaScript(이하 JS)에 Static Type 시스템을 추가한 언어

Static Type 시스템: 변수나 함수가 담을 값의 자료형을 컴파일 시점에 결정하고, 정한 규칙을 컴파일러가 검사하는 체계

**TS의 핵심**
TS는 컴파일이 끝나는 순간 결정된 타입 정보가 모두 사라지고 JS로 실행된다. 즉, TS는 JS와 비교해 별도의 언어라고 하기보다는 상위 집합 언어이기때문에 JS로 작성된 코드는 TS로도 컴파일이 유효하다.


## 2. TypeScript의 특징

### 특징
- 정적 타입 검사: 변수나 함수의 값에 타입을 명시하고, 컴파일 시점에 타입 에러를 발견하여 에러를 줄이고 안정성이 높인다.
- 객체 지향 프로그래밍: 클래스, 인터페이스, 모듈 등의 OOP 기능을 지원하여 코드의 재사용과 유지보수를 보다 효율적으로 할수있게한다.
> - 클래스: 객체를 생성하기 위해 메서드를 정의한 틀
> - 인터페이스: 객체가 가져야할 속성과 타입만을 정의하는 틀
> - 모듈: export나 import로 다른 파일에서 재사용할 수 있게 나눈 코드 단위
- IDE 지원: 코드 자동 완성, 디버깅, 문법 검사등 JS에비해 다양한 기능을 활용가능하여 생산성이 높다.
- JS의 상위 확장: JS의 기능을 사용가능하며 추가로 TS의 객체지향 패턴을 사용 가능하기때문에 기존 JS코드를 그대로 사용해도 무방하다.
- 타입 추론: 타입을 적지않아도 컴파일러가 자동으로 추론을 해준다. 대신 다른 자료형이 입력되었을때 오류를 발생한다.
- 컴파일 필요: 실행 전 JS로 변환하는 과정을 필요로한다.
- ES6+ 문법 지원: 최신 JS 표준 문법도 사용가능하다.

### 단점
- 높은 학습 곡선: JS에 익숙하더라도 TS를 사용하기위해서 익혀야할 새로운 문법과 개념이 많다.
- 과한 타입 작업: 세세하게 타입을 작성하면 오히려 개발 속도가 더 느려져 초기 구현이 늦어지는 경우가 생긴다.
- 컴파일 시간 증가: JS 변환 이전에 컴파일 과정을 거쳐야함으로 컴파일 시간이 늘어난다.
- 외부 라이브러리 타입 문제: 일부 라이브러리에서 자체 타입 정의가 없거나 부정확하면 타입 오류가 발생할 수 있다.


## 3. TypeScript, 왜 써야하는가?
- 런타임하기전에 에러를 발견할 수 있다.
- 협업 중 데이터 구조를 공유하기가 용이하다.
- React와같은 프레임 워크에서 잘못된 값을 props로 넘기는 실수를 막아준다.

## 4. TypeScript, 그러면 문제는 없는가?

**a. 타입과 실제 런타임 값의 불일지**

TS는 컴파일 시점에만 타입을 검사하기때문에 런타임 이후 외부에서 들어오는 API등의 값이 타입이 잘못 선언되어 거짓 타입을 가지게된다.

**b. 구조적 타이핑으로 인한 통과**

TS는 이름이 아니라 구조로 타입을 취급한다. 이로 인해 의미가 다른 객체가 있을때 구조만 같다면 대입이 가능해, 이러한 실수를 막지 못한다.

**c. any의 전염성**

any를 어딘가에서 쓰게되면, 그 값을 사용하는 모든 후속 코드들이 타입 검사를 받지않고 전염성을 가진다.


## 5. JavaScript와의 차이점

| 구분 | JavaScript | TypeScript |
|---|---|---|
|타입 시스템|동적 타입|정적 타입|
|오류 발견 시점|런타임(실행 중)|컴파일타임(작성 중)|
|컴파일 과정|바로 읽을 수 있음|자바스크립트로 변환 과정 필요|
|특징|코드 작성이 유연하고 빠름|타입 안정성 보장, 유지보수 용이|

## 6. .js / .ts / .jsx / .tsx의 차이점

- .js: JS로 작성되어 *타입 검사가 없고* *HTML 문법 사용이 불가*하다.
- .ts: TS로 작성되어 *타입 검사가 존재*하고 *HTML 문법 사용이 불가*하다
- .jsx: JS로 작성되어 *타입 검사가 없고* *HTML 문법 사용이 가능*하다.
- .tsx: TS로 작성되어 *타입 검사가 존재*하고 *HTML 문법 사용이 가능*하다.
