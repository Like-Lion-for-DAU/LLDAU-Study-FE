# 8주차 마이그레이션 정리 노트

## TS란
한줄로 설명하자면
js에서 타이블 붙인 언어이다.
js는 변수에 뭐가 들어가든 그다지 큰 신경을쓰지않는다.
```const x = 3```
쓰고
```x = "삼"``` 뭐 이런식으로 대충 써도 실행은 된다.
자유롭긴한데 이상하게 적어놔도 실행하기 직전까지 딱히 알려주는게 없다.
버그가 실제로 실행하는 도중에 터지게된다.

ts는 여기에 이 변수는 숫자다 이 함수는 문자열을 받는다 같은.
타입을 미리 약속하고 적게한다.
그리고 코드가 실행되기전에 그 약속이 깨졌는지 검사한다.

추가로
브라우저랑 node는 ts를 직접 읽지못한다.
그래서 ts는 실행전에 타입을 다 뗴어내고 순수 js로 변환한다.
여기선 vite가 한다.

vite.config.js는 번들링하고 실행하고
tsconfig.json은 타입 검사랑 기타등등에 쓴다

그래서 tsconfig가 이상해도 앱은 돌릴수있다
다만 타입 안전성과 에디터의 도움을 받지 못한다

에디터에서 받는 도움은 대표적으로
- 에디터 자동 완성이 좀더 정확해진다
- 잘못된 접근을 즉시 잡는다.
- 리팩터링이 안전하다

## 마이그레이션 단계
확장자 변경한거 말고도
공통으로 쓰는 데이터 모델타입인 Member이랑 Part를 types.ts로 따로 뺴서 lions.ts랑 Page.tsx가 같이 임포트된다.
함수,변수,props,상태에 타입을 붙엿다.
tsconfing,json을 추가하고 타입 검사 규칙을 넣었다

여기서 따로 파일로 뺸 이유는
Member모양을 데이터랑 화면 둘다 알아야하는데
양쪽에 똑같이 적으면 나중에 한쪽만 고치고 까먹는경우있어서
그냥 한곳에 몰아넣어서 유지보수를 편하게 하였다.

## ts문법

### 타입표기
기본 타입 표기 : 타입
값 변수 파라비터 리턴 옆에 클론으로 타입을 적는다
```
function hashSeed(str: string): number {
  let h = 2166136261;
  // ...
  return h >>> 0;
}
```
str: string써서 이 함수는 문자열만 받는다고 적어놓는다
: member 이건 숫자를 돌려준단거다

만약에 이 함수를 사용할때 hashSeed(1557)처럼 숫자를 쓰면 바로 막아준다
또한 반환된 값을 .toUpperCase() 같은 문자열 메서드를 사용할려고해도
숫자로 반환된다고 알려준다.

### 별칭 유니온 타입
type은 타입에 이름 붙이는것
|는 이것들중 하나라는 뜻인 유니온

```
export type Part = "Frontend" | "Backend" | "Design";

type FetchStatus = "idle" | "loading" | "success" | "error";
type SortKey = "default" | "name" | "part";
```

Part는 그냥 string이 아니라 이 3개중 하나만 허용하게된다.
만약 part: string이라고만하면 뭐 frontd같은 오타도 통과시키겠지만 Part유니온이면 오타가 에러로뜨게된다.

### 인터페이스
객체가 어떤 필드를 갖는지 설계한다.
```
export interface Member {
  id: number;
  name: string;
  part: Part;
  skills: string[];
  isMine: boolean;
  // ...
}
```
skills: string[]이건 문자들의 배열
part: Part 이건 위에서 만든 유니온을 필드 타입으로 재사용한거

이러면 이제 Member를 쓰는 모든곳에서 자동완성되고 오타나 없는 필드같은거 쓰려면 경고해준다.
lions.ts의 데이터도 선언해놔서 part에 딴거 적으면 바로 알수있다.

### 인터페이스 vs 타입
둘다 모양을 정의할수있지만
인터페이스는 같은 이름으로 또 선언하면 합쳐지고 타입은 유니온 교차같은 표현이 자유롭다
체감상 그냥 취향차이인것같다.

### 제네릭
함수가 어떤 타입이 올지 호출할때 정해지는경우
```
function pickRandom<T>(arr: readonly T[]): T {
  return arr[Math.floor(Math.random() * arr.length)];
}
```
여기서 T는 정해지지않은 타입 자리이다 만약 Part[]를 넣으면 T = Part가 되어서 리턴도 Part가 되고
만약 string[]을 넣으면 리턴이 string이 되게된다.

만약 제네릭 없이 arr: any[]로 하면 리턴이 아무거나되어서 ts쓰는 의미가 사라진다
제네릭을 쓰면 타입을 넣은 그대로 돌려받는다는 관계가 유지되게된다.

리엑트에서도 썼었다
```
const [members, setMembers] = useState<Member[]>(initialMembers);
const [selectedId, setSelectedId] = useState<number | null>(null);
const [following, setFollowing] = useState<Set<number>>(() => new Set());
const latestControllerRef = useRef<AbortController | null>(null);
```
useState이건 Member배열이라고 못박는거다
안적으면 초기값으로 추론되긴한데 null이 섞이면 명시해줘야한다.

### 유틸리티타임
기존 타입을 재료로 새 타입을 만든다
record pick return type

```
const PART_COLOR: Record<Part, string> = {
  Frontend: "#3fb950",
  Backend: "#58a6ff",
  Design: "#bc8cff",
};
```
Record는 키가 part의 세값 값은 string인 객체이다
세 키를 뺴먹으면 에러가 나서 파트가 추가되었는데 색을 안정한 실수를 막는다

```
function handleOf(m: Pick<Member, "name" | "email" | "website">): string
```
이건 Member전체가 아니라 이름 메일 웹사이트 세개만 쓴다.
함수가 실제로 쓰는 필드만 받겠다고 좁혀놓으면 의도를 명확하게 할수있고
아무객체나 세 필드만 맞으면 넘길수있어서 유연성도 확보가능하다.

```
const statusResetTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
```
이거 리턴타입은 환경마다 다르긴하다 브라우저는 number, 노드는 객체였나 무튼 그랬던것같다.
그래서 직접 member이라고 박아놓으면 환경따라 바뀌어서 ts가 알아서 맞는 타입 가져오게 해놓았다.

### null 유니온
값이 있을수도 없을수도 있는경우에 대비했다.

```
const [selectedId, setSelectedId] = useState<number | null>(null);
const selected = members.find((m) => m.id === selectedId) ?? null; // Member | null
```
여기서 selected는 Member거나 null이다
그냥 selected.name이라고쓰면 null이면 어쩔려고 하면서 막는다
그래서 검사로 좁혀놓고 써야한다
```
{selected && (
  <DetailModal member={selected} ... />
)}
```
안쪽에서 ts가 여기는 null아님을 인정해서 selected.id를 쓸수있게된다.

재시도 버튼에서도 같은 패턴을 사용하였다.
```onClick={() => lastFetchActionRef.current && runFetchAction(lastFetchActionRef.current)}
```
lastFetchActionRef.current가 FetchAction | null이라 &&로 null을 거른 다음에 함수로 넘긴다

null값에 호출하려다가 터지는게 js에서 좀 흔한데 ts는 강제로 실행전에 막는게 장점인듯하다.

### 언노운 케치 에러 처리
ts에서 catch(err)의 err는 any가 아니라 unknown이다.
즉 뭐가 잡혔는지 모르는 상태라 바로 못 읽는다.
```
} catch (err) {
  const e = err as { name?: string; message?: string };
  if (e?.name === "AbortError" && timedOut) { ... }
}
```
굳이 이따구로 쓰는 이유는 던져지는게 꼭 Error이라는 보장이 없다.
그래서 ts는 일단 unknown으로 막고 이건 name/message가 있는 객체야 라고 단언해서 쓰게한다.
안전하긴한데 좀 귀찮다.

### 타입 선언 as
ts한테 이건 이 타입인걸 개발자가 직접 명시하는거
쉽게 말해서 그냥 우기는거

```
// select의 value는 string으로 오지만 실제론 Part나 SortKey만 들어온다
onChange={(e) => setPartFilter(e.target.value as PartFilter)}
onChange={(e) => setSortKey(e.target.value as SortKey)}

// 폼의 part는 Part | 인데, 필수 검사를 통과했으니 Part로 단언
part: part as Part,
```
html의 <select>의 값은 무조건 string타입으로 들어온다.
근데 우리는 옵션을 Part값들로 만들었으니 실제론 part다.
근데 ts는 그걸 모르니 그냥 맞다고 우기는거다

근데 as는 남발을하면 다소 곤란해질듯하다
타입이 보장되지않는데 계속쓰면 그냥 ts쓰는 의미도없고 js쓰는것만도 못한 선택이될수도있겠다.
여기의 경우는 select옵션을 직접 통제하고 handleAdd에서 빈 part를 먼저 걸러낸뒤에만 단언한다.

### readonly
```
export const PARTS: readonly Part[] = ["Frontend", "Backend", "Design"];
```
이건 읽기만되는 Part배열이다
뭐 PARTS.push 이런거 막는다
PARTS는 고정상수라 바뀌면 안되는거라 이렇게 썼다.
그리고 pickRandom<T>(arr: readonly T[])도 readonly를 받게했다.
이러면 읽기전용 배열이든 일반 배열이든 둘다 넘길수있으니 좀더 유연하게 가능하다.

### 리엑트 타입
이벤트 핸들러나 props에는 리엑트가 주는 타입을 썼다.
```
// props 모양을 interface로 정의
interface MemberCardProps {
  member: Member;
  onOpen: (id: number) => void;
  onDelete: (id: number) => void;
}

function MemberCard({ member, onOpen, onDelete }: MemberCardProps) { ... }
```
member하나 받고 리턴없음을 void로 나타낸다
이러면 부모가 엉뚱한 함수를 넘기면 바로 알수있다.

이벤트 타입
```
function fallbackImg(e: SyntheticEvent<HTMLImageElement>, id: number): void { ... }

const input =
  (field: keyof MemberForm) =>
  (e: ChangeEvent<HTMLInputElement | HTMLTextAreaElement | HTMLSelectElement>) =>
    setForm((prev) => ({ ...prev, [field]: e.target.value }));
    ```

ChangeEvent 이건 input의 onChange이벤트 e.target.value를 안전하게 잡을수있다.
keyof MemberForm 이건 MemberForm의 키 이름들 이라는 타입임 field에 진짜 필드 이름만 허용하게된다.

반대로 기존 js안에서 인라인으로 쓴 핸들러 onClick는 타입을 안 적었다.
리엑트가 문맥으로 알아서 추론해줘서 분리된 함수만 명시해줘서 이건 깔끔했던것같다.

### 타입 임포트
타입만 가져올떄 쓰는 임포트다
```
import type { Member, Part } from "./types";
```
타입은 실행하는 시점에는 사라지는 정보이다.
근데 이걸 쓰면 이건 타입만 가져오는거고 실제 코드는 아니라는걸 명확하게해서
번들에 불필요한 요구를 안남기고 의도도 분명하게 가능하다.

## 배운점
ts는 js에 타입을 더한거다. 따라서 버그를 실행전에 잡을수있다.
핵심은 값이 어떤 모양인지 미리 약속을하고 ts가 그 약속을 꺠는 부분을 알아서 찾아준다
이번에 쓴 인터페이스 타입 유니온으로 모양을 정의해서 제네릭 유틸리티 아입으로 재사용하고 null로 좁히고 unknown as로 안전하게 다루었다
타입은 좀 귀찮긴한데 내가 잘 실수하는 부분의 버그를 잘 봐줘서 나쁘지않은것같다.
